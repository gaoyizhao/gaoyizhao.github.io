memmove和memcpy的区别和strcpy实现

1.memmove
函数原型：

void memmove(void dest,const void *source,size_t count)

返回值说明：返回指向dest的void *指针
参数说明：dest,source分别为目标串和源串的首地址。count为要移动的字符的个数
函数说明：memmove用于从source拷贝count个字符到dest,如果目标区域和源区域有重叠的话,memmove能够保证在源串覆盖之前将重叠区域的字节拷贝到目标区域中。

2.memcpy
函数原型：

void memcpy(void dest,const void *source,size_t count)

返回值说明：返回指向dest的void *指针
函数说明：memcpy功能和memmove相同，但是memcpy中dest和source中的区域不能重叠，否则会出现未知错误结果。

3.两者区别

函数memcpy()：从source指向的区域向dest指向的区域复制count个字符，如果两组数重叠，不定义该函数的行为。
函数memmove()：两区域重叠，复制可以正确进行。

memcpy函数假设要复制的内存区域不存在重叠，若能保证进行复制操作的内存区域没有任何重叠，可以直接用memcpy;
memmove函数绝对保证复制正确性，即使内存区域重叠。
但是memcpy效率比memmove高一些。

1.教科书里的单例模式

2.懒汉与饿汉

单例大约有两种实现方法：懒汉与饿汉。
懒汉：顾名思义，不到万不得已，不会去实例化类，也就是第一次用到类实例的时候才会去实例化，上边的经典方法为懒汉实现；
饿汉：饿了肯定要饥不择食。所以在单例类定义的时候就进行实例化。

特点与选择：
①由于要进行线程同步，在访问量比较大，或者可能访问的线程比较多时，采用饿汉实现，可以实现更好的性能。只是以空间换时间。
②在访问量比较小时，采用懒汉实现。这是以时间换取空间。

STL六大组件简介

STL六大组件简介

一、STL简介

（一）、泛型程序设计
STL（Standard Template Library），即标准模板库，是一个高效的C++程序库。
泛型编程，将程序写的尽可能通用，将算法从数据结构中抽象出来，成为通用C++的模板。
（二）、STL组件
容器（Container）：各种基本数据结构。
适配器（Adapter）：可改变容器、迭代器或仿函数接口的一种组件。
算法（Algorithm）：各种基本算法如sort、search…等。
迭代器（iterator）：连接容器和算法。
仿函数（Function object）：也叫作函数对象。
分配器（Allocator）：空间配置器，分配内存。

二、六大组件简介
1、容器：

容器类是容纳、包含一组元素或元素集合的对象。
七中基本容器：
向量（vector）、双端队列（deque）、list、set、multiset、map、multimap。

vector：动态数组，堆中分配内存，元素连续存放。访问元素O(1)。末尾元素操作最快。
在中间或开始处进行添加或者删除元素需要移动内存，如果元素是结构体或者类，移动的同时会进行构造和析构，影响性能。

list:类似链表，元素在堆中存放，分配元素增加空间，删除元素，释放空间。可快速在所有地方添加删除元素。访问最开始和最末尾元素o(1)。
访问中间元素o(n)。

deque:双端队列，在堆中开辟内存。并且分为好几个堆，堆1，堆2，堆3…每个堆保存元素，堆和堆之间用指针指向。在开始和最后添加删除元素最快，具有较高的随机访问速度，并且提供随机访问方法，可用[]访问任意元素，但是访问速度比不上vector,因为要内部处理堆跳转。
deque和vector都会预留空间，deque在前面和后面添加元素不需要移动数据。

set/map
相同点：所有元素根据元素的键值自动被排序，set 和 map两者的各种操作，都只是转而调用RB-Tree的操作行为，不过两者都不允许两个元素有相同的键值。
不同点：set的元素不像map元素那样可以同时拥有实值（value）和键值（key）,map的所有元素都是pair,pair的第一个元素被视为键值（key）,第二个元素被视为实值（value）。set的键值就是实值，实值就是键值，只有一个元素。

multiset/multimap ：特性以及用法和set/map完全相同，唯一的差别在于其允许键值重复，即所有的插入操作基于RB-Tree的insert_equal()而非insert_unique()。

hash_set/hash_map/hash_multiset/hash_multimap

hash_set/hash_map,两者的一切操作都是基于hashtable之上，hash_set
和set一样，实值就是键值，键值就是实值，而hash_map和map一样，每个元素同时拥有一个实值（value）和键值（key）。由于hash_set和hash_map都是基于hashtable之上，所以不具备自动排序功能。hashtable没有自动排序功能。

hash_multiset/hash_multimap的特性与上面的multiset/multimap完全相同，唯一的差别
就是底层实现机制是hashtable,其元素不会自动排序。

总结：
set/map/multiset/multimap都是基于RB-Tree,所以有自动排序功能。
hash_set/hash_map/hash_multiset/hashmultimap都是基于hashtable,不含自动排序功能。至于前面加个前缀multi无非是允许键值重复而已。

C++11 新特性：
std:tr1::unordered_map是无序哈希表，但是操作效率比std::map,std::hash_map/hash_multiset/hash_multimap
__gnu_cxx::hash_map 都要高，可以研究一下。

2、 迭代器

迭代器iterator,用来在一个对象群集的元素上进行遍历。迭代器的好处是，为所有容器提供了一组很小的公共接口。迭代器以++进行累进，*操作进行解引用，类似于指针，可视为一种smart pointer。

3、算法

算法，用来处理集群内的元素。可将同一算法，应用于任意容器，因为所有的容器迭代器都提供一致的接口。

4、适配器

适配器是一种接口类
为已有的类型提供接口。
目的是简化，约束，使之安全，隐藏或者改变被修改类提供的服务集合。
三种类型适配器：
容器适配器：用来扩展7中基本容器，它们和顺序容器相结合构成栈、队列和优先队列容器。
迭代器适配器（反向迭代器、插入迭代器、IO流迭代器）。
函数适配器（函数对象适配器、成员函数适配器、普通函数适配器）。

5、函数对象（仿函数）

1、一个行为类似函数的对象，可以没有参数，也可以带有若干参数。
2、任何重载了调用运算符operator()的类的对象都满足函数对象的特征。
3、STL中也定义了一些标准的函数对象，如果以功能划分，可以分为算数运算、关系运算、逻辑运算三大类。为了调用这些标准函数对象，需要包含头文”functional”。

6、SGI 空间适配器

负责空间配置与管理
当申请内存大于128byte时，启动一级空间适配器malloc直接向系统的堆空间分配，
若申请的内存小于128byte时，启动二级空间适配器，从一个预先分配好的内存池中取一块内
存交付给用户，这个内存由16个不同大小（8的倍数，8~128bytes）的空闲列表组成，allocator会根据申请内存的大小（round up为8的倍数）从对应的空闲块列表取表头块给用户。
优点：小对象的快速分配.
避免内存碎片的生成。

可执行程序加载到内存的过程

Posté le 2017-06-16

可执行程序加载到内存的过程

在Linux中，程序的加载，涉及到两个工具，linker和loader。Linker主要涉及动态链接库的使用，Loader主要涉及软件加载。

1、exec执行一个程序。

2、elf是现在非常流行的可执行文件格式，其为程序运行规划了两个段：一个段是可执行的代码段，它是只读，可执行；另一个是数据段，它是可读写，不能执行。

3、Loader会启动，通过mmap()系统调用，将代码段数据段映射到内存，即为其分配了虚拟内存，这时候不占用物理内存；只有当程序执行到相应的地方，内核才为其分配物理内存。

4、Loader会去查找该程序依赖的链接库，首先该链接库是否被映射进内存中，未映射，则使用mmap，将代码段数据段映射到内存中；否则只是将其加入进程的地址空间。这样和glibc等库的内存地址空间完全一样。

glibc是gnu发布的libc库，即c运行库。是linux系统中最底层的api，几乎其它任何运行库都会依赖于glibc。glibc除了封装linux操作系统所提供的系统服务外，它本身也提供了许多其它一些必要功能服务的实现。由于glibc囊括了几乎所有的UNIX通行的标准，可见其内容包罗万象。而就像其他的UNIX系统一样，其内含的档案群分散于系统的树状目录结构中，像一个支架一般撑起整个作业系统。

因此，一个2M的程序，执行时，并不意味为其分配2M的物理内存，这与其运行的代码量，与其所依赖的动态链接库有关。

运行过程中链接动态链接库 与 编译过程中链接动态链接库 区别

调用动态链接库有两种方法：
一种是编译的时候，指明所依赖的动态链接库，这样loader可以在程序启动的时候，将所有的动态链接库映射到内存中；
一种是在运行过程中，通过dlopen和dlfree的方式加载动态链接库，动态将动态链接库加载到内存。

这两种方式，从编程角度讲，第一种最方便，效率上差别不大，但在内存使用上有差别。

第一种方式，一个库的代码，只要运行过一次，便会占用物理内存，之后即使再也不使用，也会占用物理内存（相当于扩大程序代码段），直到进程终止；
第二种方式，库代码占用的内存，可以通过dlfree的方式，释放掉，返回给物理内存。适用于偶尔而调用各种库的进程。

内联函数

1. inline函数

在函数声明或定义中函数返回类型前加上关键字inline，即把函数指定为内联。
inline函数对编译器而言必须是可见的，以便能够在调用点展开该函数。与非inline函数不同的是，inline函数必须在调用该函数的每个文本文件中定义。当然，对于同一程序的不同文件，如果inlin函数出现的话，其定义必须相同。否则定义不相同，程序将会有未定义的行为。故建议将内联函数定义到头文件中。每个调用该inline函数的文件都包含该文件。
这种方法保证每个inline函数只有一个定义，且程序无需复制代码，并且不可能在程序的生命期中引起无意义的不匹配的事情。

2. inline函数的编程风格

关键字inline必须与函数定义提放在一起才能成为内联，仅将inline放在函数声明前面不起任何作用。

故,inline是一种“用于实现的关键字”，而不是一种“用于声明的关键字”。一般说用户可以阅读函数的声明，但是看不到函数的定义。尽管大多数教科书内联函数的声明、定义体前面都加了inline关键字，但我认为inline不应该出现在函数的声明中。这个细节体现了高质量C++/C程序设计风格的一个基本原则：声明与定义不可以混为一谈，用户没必要知道具体的函数定义，没有必要知道函数是否内联。

3. 慎用内联

内联能提高函数执行效率，但是内联函数是以代码膨胀（复制）为代价，仅仅省去函数调用的开销，从而提高函数的执行效率。如果执行函数体内代码的时间，相比于函数调用的开销大则效率的收获会很小。另一方面，每一处内联函数的调用都要复制代码，将使程序的总代码量增大，消耗更多的内存空间。以下情况不宜使用内联：

（一）若果函数体内代码较长，使用内联将导致内存消耗代价较高。
（二）如果函数体内出现循环，那么执行函数体内代码的时间比函数调用开销大。
类的构造函数和析构函数容易让人误解使用内联更有效。要当心构造函数和析构函数可能会隐藏一些行为如“偷偷地”执行了基类或成员对象的构造函数和析构函数，导致效率降低。故不要随意将构造函数和析构函数定义体放在类声明中。

一个好的编译器将会根据函数的定义体，自动取消不值得的内联（进一步说明inline不应该出现在函数声明中）

C++语言支持函数内联，目的是为了提高函数执行效率（速度）。

在C程序中，可以用宏代替提高执行效率。宏代码本身不是函数，但是使用起来像函数。预处理器用复制宏代码的方式代替函数调用，省去了参数压栈、生成汇编语言的CALL调用、返回参数。执行return等过程，从而提高了速度。

使用宏代码最大的缺点是容易出错，预处理器在复制宏代码时常常产生意想不到的边际效果。

对于C++而言，使用宏代码还有另一缺点：无法操作类的私有数据成员。

C++内联函数工作：

对于任何内联函数，编译器在符号表里放入函数声明（包括名字 参数类型 返回值类型）。
若编译器没有发现内联函数存在错误，则该函数的代码也将放入符号表中。
在调用一个内联函数时，编译器首先检查调用是否正确（进行类型安全检查，或者进行自动类型转换，当然所有函数都一样）。如果正确，内联函数的代码直接替换函数调用，于是省去了函数调用的开销。

这个过程与预处理显著不同，预处理器不能进行类型安全检查，或者进行自动类型转换。假如内联函数是成员函数，对象的地址会被放在合适的地方，这些预处理器都办不到。

C++语言的函数内敛机制即具备宏代码的效率，又增强了安全性，而且可以自由操作类的数据成员。
故在C++程序中，应该用内联函数取代所有宏代码。”断言assert”是唯一的例外。

assert仅在Debug版本起作用，用于检查不应该发生的情况。
为了不在程序的Debug和Release版本中引起差别，assert不应该产生任何副作用。
如果assert是函数，由于函数调用会引起内存代码的变动，那么将导致Debug版本和Release版本存在差异。
故assert不是函数，而是宏。
